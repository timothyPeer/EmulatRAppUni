# Alpha Memory Subsystem — Corrected Layering & Contract

 

## Purpose of This Document

 

This section defines the **authoritative responsibility boundaries** for the Alpha emulator memory subsystem and corrects an architectural violation where RAM/MMIO checks leaked into the wrong layer.

 

This document is normative: **future code must conform to these rules**.

 

---

 

## High-Level Memory Stack (Authoritative)

 

```

AlphaCPU / DMA / Devices

       │

       ▼

AlphaSystemMemory

       │

       ▼

GuestMemory        ← Physical Address SpaOk. e Authority

       │

       ├── SafeMemory   (RAM backing store)

       └── MMIOManager  (Devices)

```

 

---

 

## Layer Responsibilities (Corrected & Final)

 

1. `SafeMemory` — Raw Physical RAM Backing Store

 

**Role**

 

* Owns contiguous RAM backing storage

* Performs byte/word/quad loads and stores

* Enforces:

 

 * bounds

 * alignment

 * atomicity

 * endianness

 

**Explicitly MUST NOT**

 

* Determine whether an address is RAM or MMIO

* Know about:

 

 * GuestMemory

 * AlphaSystemMemory

 * CPU privilege

 * platform address maps

 

**Invariant**

 

> Every access reaching SafeMemory is **already known to be valid RAM**.

 

---

 

2. `GuestMemory` — Physical Address Space Authority

 

**Role**

 

* Single source of truth for **physical address routing**

* Owns the platform memory map:

 

 * RAM ranges

 * MMIO apertures

 * device BARs

* Dispatches accesses to:

 

 * `SafeMemory` (RAM)

 * `MMIOManager` (MMIO)

 

**Responsibilities**

 

* Decide **RAM vs MMIO**

* Validate RAM access ranges

* Reject illegal physical addresses

* Remain platform-configurable (SRM, VMS, Tru64, Linux, custom)

 

**Invariant**

 

> No code above or below `GuestMemory` may independently decide RAM vs MMIO.

 

---

 

3. `AlphaSystemMemory` — CPU-Facing Memory Semantics

 

**Role**

 

* Implements architectural semantics for memory access:

 

 * MMU translation

 * ITB/DTB lookup

 * privilege checks

 * alignment faults

 * access size rules

 * fault synthesis

 

**Delegation Rule**

 

* After producing a **final physical address**, all access **must pass through `GuestMemory`**.

 

**Explicitly MUST NOT**

 

* Query RAM/MMIO mappings

* Call `SafeMemory` directly

* Duplicate platform address map logic

 

**Invariant**

 

> `AlphaSystemMemory` produces *meaning*, not *routing*.

 

---

 

## The Bug (What Was Wrong)

 

Observed Violation

 

`AlphaSystemMemory` (or a lower helper) performed checks such as:

 

* “Is this RAM?”

* “Is this GuestMemory?”

* “Does this fall into SafeMemory?”

 

This resulted in:

 

* duplicated policy

* inconsistent routing

* broken abstraction layering

 

---

 

## The Fix (Required Changes)

 

Rule 1 — Single Routing Decision

 

**Only `GuestMemory` may decide:**

 

```text

Physical Address → RAM or MMIO

```

 

---

 

Rule 2 — SafeMemory Assumes Valid RAM

 

All functions in `SafeMemory` assume:

 

* address is RAM

* address is already validated

 

**No platform logic allowed.**

 

---

 

Rule 3 — AlphaSystemMemory Delegates Routing

 

# Correct Pattern

 

```cpp

// AlphaSystemMemory

PhysicalAddress pa = translateVA(...);

 

// GuestMemory decides

guestMemory->access(pa, size, type, buffer);

```

 

# Forbidden Pattern

 

```cpp

// ❌ DO NOT DO THIS

if (isRAM(pa)) {

   safeMemory->read(pa, ...);

} else {

   mmio->write(pa, ...);

}

```

 

---

 

## Corrected Call Flow (Canonical)

 

Load / Store Path

 

```

AlphaCPU

 ↓

AlphaSystemMemory

 - MMU translate

 - permission check

 - alignment check

 ↓

GuestMemory

 - RAM vs MMIO routing

 ↓

SafeMemory OR MMIOManager

```

 

---

 

## Architectural Invariants (Must Hold)

 

1. **Exactly one RAM/MMIO decision**

2. **SafeMemory never rejects MMIO**

3. **AlphaSystemMemory never knows platform layout**

4. **GuestMemory is the physical authority**

5. **No circular knowledge between layers**
