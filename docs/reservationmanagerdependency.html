ReservationManager

 

Structural Requirements

RM-REQ-001:

Provide per-CPU reservation tracking for LL/SC instructions.

RM-REQ-002:

Track the reserved physical address, aligned to a “granule” (32B EV4 / 64B EV6).

RM-REQ-003:

One active reservation per CPU at any time.

RM-REQ-004:

ReservationStorage must be sized to numCpus (or MAX_CPUS in default mode).

RM-REQ-005:

Provide (optional) timestamp metadata per CPU reservation.

 

 

LL/SC Functional Requirements

RM-REQ-010:

reserve(cpuId, pa) must record the reservation atomically.

RM-REQ-011:

tryStoreConditional(cpuId, pa) must succeed only if: The address is in the same granule as the stored reservation. The reservation is valid at the moment of check.

RM-REQ-012:

Successful STx_C must invalidate the reservation.

 

Write-Invalidation Requirements

RM-REQ-020:

Any write by any CPU to the granule must invalidate all other CPUs’ reservations.

RM-REQ-021:

Any DMA / device memory write into RAM (via SafeMemory or MMIO DMA path) must trigger invalidation.

RM-REQ-022:

Invalidation must be performed under write-lock and be atomic with respect to CPU reservations.

 

Callbacks and Notifications

RM-REQ-030:

Provide a callback (m_onInvalidated) invoked whenever a reservation is invalidated.

RM-REQ-031:

Callback MUST be invoked after releasing the internal write-lock.

RM-REQ-032:

Allow EmulatorManager to attach a callback that sends a reservation-break IPI via IRQController.

 

Context-Switch Requirements

RM-REQ-040:

Reservations MUST be cleared when a CPU: Enters PAL code, Exits PAL code, Changes ASN, Changes PCB or kernel thread, Performs REI

RM-REQ-041:

Provide a method clearReservation(cpuId) suitable for PAL handler usage.
