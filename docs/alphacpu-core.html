# Chapter 12 – AlphaCPU Core

 

## 12.1 Purpose of This Chapter

 

This chapter defines the **AlphaCPU core object** and its responsibilities within EmulatR.

 

AlphaCPU is the **unit of execution**. It owns:

 

* The processor run loop

* Pipeline state

* Architectural registers

* Per-CPU execution context

* Interaction with SMP, interrupts, and PAL

 

This chapter does **not** redefine architectural rules (those are in Group 1).

Instead, it explains **how those rules are enforced in code**.

 

---

 

## 12.2 AlphaCPU as the Unit of Execution

 

Each AlphaCPU instance represents **one physical Alpha processor**.

 

Key properties:

 

* One AlphaCPU = one hardware CPU

* Each AlphaCPU runs independently

* All AlphaCPUs execute concurrently in SMP

* AlphaCPU owns *all* per-CPU state

 

There is no shared CPU object and no master CPU.

 

---

 

## 12.3 Ownership Model

 

### 12.3.1 AlphaCPU Owns

 

AlphaCPU directly owns or controls:

 

* AlphaPipeline

* Register banks (integer, FP)

* Processor status (PS, IPL)

* TLBs (ITB / DTB)

* Write buffers

* LL/SC reservation state (via GuestMemory hooks)

* FaultDispatcher (per-CPU view)

* PalBox execution context

* CPU-local interrupt state

 

No other subsystem may mutate this state directly.

 

---

 

### 12.3.2 AlphaCPU Does *Not* Own

 

AlphaCPU does **not** own:

 

* GuestMemory (shared)

* SafeMemory (shared)

* Devices or MMIO

* DeviceManager

* MMIOManager

* SMPManager

* Global interrupt controller

 

These are accessed via defined interfaces only.

 

---

 

## 12.4 CPU Lifecycle

 

### 12.4.1 Creation

 

AlphaCPUs are created by **SMPManager**.

 

During creation:

 

* CPUId is assigned

* Architectural state is initialized

* Pipeline is reset

* TLBs are cleared

* PAL mode is inactive

* Interrupts are masked

* PC is set to reset vector

 

No execution occurs during construction.

 

---

 

### 12.4.2 Startup

 

Execution begins when:

 

* SMPManager starts the CPU run loop

* Initial PC is valid

* Reset state is complete

 

Each AlphaCPU enters its own run loop.

 

---

 

### 12.4.3 Shutdown

 

AlphaCPU exits its run loop when:

 

* HALT instruction retires

* Fatal exception occurs

* External shutdown signal is received

* Simulation ends

 

Shutdown is graceful:

 

* Pipeline drains or flushes

* Device state remains intact

* Other CPUs continue unless directed otherwise

 

---

 

## 12.5 The AlphaCPU Run Loop

 

### 12.5.1 Role of the Run Loop

 

The run loop is the **clocked execution engine**.

 

Each iteration represents **one hardware cycle**.

 

Responsibilities per cycle:

 

1. Pre-cycle checks (halt, interrupts, exceptions)

2. Pipeline advancement

3. Barrier release evaluation

4. Exception / interrupt delivery

5. Housekeeping and statistics

 

---

 

### 12.5.2 Run Loop Invariants

 

The run loop must enforce:

 

* One cycle per iteration

* Deterministic ordering

* No mid-instruction interruption

* No speculative privilege leakage

* Precise exception semantics

 

The run loop is the *only* place where execution advances.

 

---

 

## 12.6 Pipeline Integration

 

AlphaCPU owns the **AlphaPipeline** and drives it explicitly.

 

Responsibilities:

 

* Advance pipeline stages

* Handle stalls and flushes

* Coordinate serialization

* Commit architectural state in WB

* Restart fetch after control flow changes

 

AlphaCPU does **not** execute instructions directly — it delegates execution to the pipeline and grains.

 

---

 

## 12.7 Interaction with FaultDispatcher

 

AlphaCPU is responsible for:

 

* Polling pending faults and interrupts

* Determining delivery timing

* Initiating pipeline flushes

* Entering PAL on delivery

 

Fault detection may occur anywhere, but **delivery is centralized in AlphaCPU**.

 

---

 

## 12.8 Interrupt Handling Responsibility

 

AlphaCPU:

 

* Samples interrupt state during pre-cycle phase

* Applies masking and IPL rules

* Selects highest-priority pending interrupt

* Initiates PAL entry when delivered

 

Interrupts never interrupt instruction execution mid-cycle.

 

---

 

## 12.9 PAL Integration

 

AlphaCPU enforces PAL semantics by:

 

* Serializing CALL_PAL

* Clearing LL/SC reservations on PAL entry

* Capturing register-context snapshots

* Blocking speculative execution

* Enforcing HW_REI restore semantics

 

PAL execution is coordinated, not special-cased.

 

---

 

## 12.10 SMP Awareness

 

Each AlphaCPU:

 

* Knows its CPUId

* Participates in SMP coordination

* Receives IPIs as interrupts

* Performs local TLB invalidation

* Enforces per-CPU privilege boundaries

 

AlphaCPU never directly modifies another CPU’s state.

 

---

 

## 12.11 Timing and Determinism

 

AlphaCPU guarantees:

 

* Deterministic execution per CPU

* Deterministic interaction ordering

* Explicit visibility boundaries

 

All nondeterminism is **intentional and observable**.

 

---

 

## 12.12 Error Handling Policy

 

AlphaCPU responds to errors by:

 

* Raising precise exceptions

* Entering PAL

* Halting execution if unrecoverable

 

Undefined behavior is not permitted.

 

---

 

## 12.13 Summary

 

AlphaCPU is:

 

* The sole execution authority

* The owner of per-CPU architectural state

* The driver of the pipeline

* The arbiter of exceptions and interrupts

* The enforcer of privilege boundaries

 

Nothing executes unless AlphaCPU allows it.
