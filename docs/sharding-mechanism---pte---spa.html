Sharding Mechanism - PTE - SPAMShardManager

What is Sharding?

-----------------

Sharding is the process of partitioning a large set of cache (TLB/PTE) entries into smaller, semi-independent subsets called "shards."  Each shard operates like a mini-cache, responsible for a subset of addresses, CPUs, realms, or size classes.

 

In the Alpha PTE subsystem, the SPAMShardManager implements sharding along several axes:

•**By CPU**: Each logical CPU (core/thread) receives its own slice of the TLB cache.

•**By Realm**: Separate shards are maintained for instruction (ITB) and data (DTB) translation.

•**By Size Class**: If enabled (ShardBySize=true), entries can be grouped by page size (e.g., 8K, 64K).

•**By Bucket Index**: Each shard subdivides its set into hash buckets for fast lookup.

 

How Sharding Works

------------------

•The system allocates a multidimensional array of shards (e.g., [CPU][Realm][SizeClass][BucketIndex]).

•Each CPU/thread works primarily with its own shard(s), reducing contention.

•When a translation (TLB) operation occurs, SPAMShardManager computes which shard/bucket should handle the VA+ASN based on the CPU, realm (I/D), and size class.

•Entries are inserted, looked up, and invalidated only within their target shard, making operations more scalable and cache-efficient.

 

Configuration Example:

    // 4 CPUs, 2 realms (ITB/DTB), 2 size classes, 1024 buckets:

    m_buckets.resize(4); // CPUs

    for (int c = 0; c < 4; ++c)

        for (int r = 0; r < 2; ++r)

            for (int s = 0; s < 2; ++s)

                m_buckets[c][r][s].resize(1024);

 

Benefits of Sharding

--------------------

1. **Scalability**: Each CPU or thread has its own subset of entries, minimizing contention and lock overhead in multicore (SMP) systems.

2. **Parallelism**: TLB fills, lookups, and invalidations can occur in parallel on different CPUs without interference.

3. **Cache Locality**: Per-CPU shards improve CPU cache usage, as each thread tends to access its own recently used entries.

4. **Lock-Free or Fine-Grained Locking**: With per-shard or per-bucket synchronization, global locks are not needed.

5. **Isolation**: One CPU or realm's flush or invalidation affects only its shard, reducing unnecessary global invalidations.

 

Requirements and Design Considerations

--------------------------------------

•**Shard Dimensioning**: Must allocate enough shards to match the CPU count, realms (usually 2: ITB and DTB), and size classes required by the hardware and OS.

•**Bucket Count**: Each shard subdivides into hash buckets for fast lookup; bucket count should be a power of two for efficient hashing.

•**ASN Management**: Each shard maintains its own Address Space Number (ASN) epochs for efficient per-ASN invalidation.

•**Concurrency**: If CPUs/threads operate in parallel, per-shard or per-bucket data must be thread-safe (e.g., atomic counters, QAtomicInteger, per-bucket locks).

•**Consistency**: Broadcast invalidation (e.g., TBIA, TBIAP) may need to coordinate across all shards, such as via IPI messages in SMP.

 

Notes on Implementation

-----------------------

•The SPAMShardManager uses Qt containers (QVector, etc.) and static arrays for fast access.

•Bucket index calculation is typically based on VA, realm, and size class (hashing).

•Per-CPU ASN epochs enable instant invalidation of all entries for a given ASN (no scanning required).

 

Example (Index Calculation):

    // Tag struct contains vpn, realm, sizeClass.

    // Hash tag to pick a bucket inside a CPU/realm/size shard.

    static inline unsigned bucketIndex(const Tag& t) noexcept {

        quint64 x = t.vpn ^ (quint64(t.realm) << 1) ^ (quint64(t.sizeClass) << 2);

        x ^= x >> 33; x *= 0xff51afd7ed558ccdULL;

        x ^= x >> 33; x *= 0xc4ceb9fe1a85ec53ULL;

        x ^= x >> 33;

        return unsigned(x) & (BucketCount - 1);

    }

 

Caveats and Best Practices

--------------------------

•Make sure all CPUs and realms have correctly dimensioned shards on initialization.

•Sharding should match the parallelism and address space requirements of your target hardware.

•Fine-grained sharding is more important as CPU count increases (e.g., >2 CPUs).

•For uniprocessor simulation, sharding can be simplified, but keeping the structure aids future SMP support.

 

Summary

-------

Sharding is a core scalability and efficiency feature in the PTE/TLB subsystem, enabling per-CPU, per-realm, and per-size class caches that work in parallel and with minimal contention.
