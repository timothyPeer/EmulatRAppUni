# Chapter 21 – Debugging, Tracing, and Determinism

 

## 21.1 Purpose and Philosophy

 

This chapter defines how ASA-EmulatR enables **systematic debugging**, **high-fidelity tracing**, and **repeatable deterministic execution** across single-CPU and SMP configurations.

 

> **Core Principle:**

> An emulator that cannot explain *why* something happened is not finished.

 

ASA-EmulatR treats debugging and determinism as **first-class architectural features**, not optional tooling layered on afterward.

 

---

 

## 21.2 Determinism as a Design Goal

 

### What Determinism Means Here

 

Determinism means:

 

* Given the same initial state

* With the same inputs

* And the same configuration

 

→ **The emulator produces the same architectural outcomes every time.**

 

This includes:

 

* Instruction retirement order

* Exception delivery order

* Interrupt timing (relative to cycles)

* DMA visibility

* SMP interactions

 

Determinism is required for:

 

* Debugging

* Regression testing

* Architectural validation

* Long-running SMP correctness

 

---

 

## 21.3 Sources of Nondeterminism (Identified and Controlled)

 

ASA-EmulatR explicitly identifies potential nondeterminism sources:

 

| Source             | Control Strategy                 |

| ------------------ | -------------------------------- |

| Device timing      | Deterministic scheduling modes   |

| Interrupt arrival  | Sampled at fixed run-loop points |

| DMA completion     | Explicit completion events       |

| SMP race windows   | Architectural serialization      |

| Host thread timing | Emulator-controlled ordering     |

 

> **Invariant:**

> Host scheduling must never influence architectural outcome.

 

---

 

## 21.4 Deterministic Scheduling Model

 

### CPU Execution

 

Each CPU:

 

* Executes one architectural cycle per run-loop iteration

* Advances pipeline stages in a fixed order

* Samples interrupts only at defined boundaries

 

No instruction is:

 

* Preempted mid-execution

* Interrupted mid-pipeline stage

* Reordered relative to architectural rules

 

---

 

### Device Execution

 

Devices:

 

* Execute asynchronously in *emulator time*

* Signal completion via queued events

* Deliver interrupts only at sampling points

 

Optional deterministic mode:

 

* Device progress advances only when CPU cycles advance

* Eliminates host timing variance

 

---

 

## 21.5 Tracing Architecture (Observational Only)

 

Tracing in ASA-EmulatR is **strictly observational**.

 

> **Rule:**

> Tracing must never affect architectural behavior.

 

All trace hooks:

 

* Observe committed state

* Do not inject ordering

* Do not alter timing

* Do not allocate memory in hot paths

 

---

 

## 21.6 Instruction Tracing

 

Instruction tracing may record:

 

* PC at fetch

* Decoded instruction

* Grain identity

* Operand values (optional)

* Result values

* Retirement cycle

 

Granularity is configurable:

 

* Fetch-level

* Execute-level

* Retirement-level

 

Tracing may be:

 

* Continuous

* Conditional (PC range, opcode, CPU)

* Triggered by events

 

---

 

## 21.7 Pipeline and Stall Tracing

 

Pipeline tracing can observe:

 

* Slot occupancy

* Stall reasons

* Barrier engagement and release

* Hazard detection

* Flush events

 

This is critical for:

 

* Diagnosing deadlocks

* Verifying barrier correctness

* Understanding performance behavior

 

---

 

## 21.8 Exception and Fault Tracing

 

FaultDispatcher integrates deeply with tracing.

 

Traceable events include:

 

* Fault creation

* Priority arbitration

* Deliverability decisions

* PAL vector selection

* HW_REI restoration

 

Each exception is traceable to:

 

* Exact faulting instruction

* Exact architectural state

* Exact cycle

 

---

 

## 21.9 Interrupt and IPI Tracing

 

Interrupt tracing records:

 

* IRQ assertion

* Masking decisions

* Delivery cycle

* PAL entry

* Handler completion

 

IPI tracing includes:

 

* Sender CPU

* Target CPU(s)

* Payload type

* Acknowledgment timing

 

This is essential for SMP debugging.

 

---

 

## 21.10 Memory and LL/SC Diagnostics

 

Memory tracing supports:

 

* Load/store address and size

* Write buffer enqueue/drain

* Barrier-forced drains

* LL/SC reservation set/invalidate

* Reservation failure diagnostics

 

False sharing and livelock scenarios are explicitly observable.

 

---

 

## 21.11 Deterministic Replay

 

ASA-EmulatR supports **deterministic replay** by recording:

 

* External inputs

* Device events

* Interrupt injections

* Timing metadata

 

Replay guarantees:

 

* Identical execution

* Identical fault timing

* Identical SMP interaction

 

This enables:

 

* Binary-level bug reproduction

* Cross-version validation

* Regression testing

 

---

 

## 21.12 Breakpoints and Watchpoints

 

### Breakpoints

 

Supported breakpoint types:

 

* PC breakpoints

* Instruction-class breakpoints

* PAL entry breakpoints

* Exception breakpoints

 

Breakpoints:

 

* Trigger only at architectural boundaries

* Never mid-instruction

* Preserve precise semantics

 

---

 

### Watchpoints

 

Watchpoints observe:

 

* Memory reads

* Memory writes

* Register writes

* IPR changes

 

They are observational and do not alter ordering.

 

---

 

## 21.13 Debugger Interaction Model

 

When a debug event triggers:

 

* Pipeline drains to a precise point

* Architectural state is frozen

* Debugger may inspect state

* No partial instruction effects exist

 

Single-step mode:

 

* Executes exactly one instruction

* Delivers a debug trap afterward

* Preserves precise exception semantics

 

---

 

## 21.14 SMP Debugging Considerations

 

In SMP:

 

* Each CPU has independent trace streams

* Global ordering can be reconstructed via timestamps

* Cross-CPU events are correlated explicitly

 

There is **no implicit global serialization** for debugging.

 

---

 

## 21.15 Performance vs Observability

 

Tracing is:

 

* Compile-time configurable

* Runtime filterable

* Designed to be removable from hot paths

 

You can:

 

* Run fully instrumented builds

* Run near-production builds

* Toggle determinism and tracing independently

 

---

 

## 21.16 Architectural Invariants (Normative)

 

1. Debugging must not alter behavior

2. Tracing is observational only

3. Determinism is enforceable

4. Interrupt timing is sampled, not injected

5. Replay must be exact

6. SMP behavior must be reproducible

7. Debug events occur only at precise boundaries

 

Violating any invariant invalidates emulator credibility.
