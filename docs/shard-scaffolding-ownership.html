Shard Scaffolding Ownership

Class Ownership and Invalidation Coordination in SMP

===================================================

 

Ownership Hierarchy

-------------------

In a scalable SMP Alpha AXP emulator, the cache/TLB managers responsible for sharded SPAM entries should be owned and orchestrated at the system or SMP “manager” level, not per-CPU. A typical class hierarchy and ownership might look like:

 

    SMPManager (or EmulatorManager/SystemManager)

        └─ owns → SPAMShardManager (or SPAMShardIPRManager)

            └─ manages all TLB/PTE shards for all CPUs/realms/size classes

        └─ owns → AlphaCPU[0..N-1]

            └─ each AlphaCPU references its local slice(s) of the SPAM manager

 

Consistency and Broadcast Invalidation

--------------------------------------

•System-wide invalidations (TBIA: TLB Invalidate All, TBIAP: TLB Invalidate ASN/Process) must operate **across all shards**, not just a single CPU’s cache.

•In a real SMP system, this is typically achieved via Inter-Processor Interrupt (IPI) messages: the SMPManager broadcasts an invalidation request to all CPUs.

•Each AlphaCPU, upon receiving the IPI, calls the appropriate invalidation routine in its associated shard or slice of the SPAMShardManager.

•The SPAMShardManager provides methods (e.g., invalidateASN_AllShards(), tbia()) that iterate over all shards/CPUs and perform the invalidation.

 

Benefits of Central Ownership

-----------------------------

•Ensures a **single source of truth** for all TLB entries in the system.

•Simplifies coordination of SMP-wide events (flush, shootdown, broadcast).

•Decouples per-CPU logic from global cache management, enabling efficient broadcast, stats gathering, and policy switching.

•Allows per-CPU AlphaCPU objects to interact with their TLB slice without duplicating state.

 

Implementation Example

----------------------

Example pseudocode for SMP-aware invalidation:

 

Step 1: Define or Alias the Manager Type With All Template Parameters

--------------------------------------------------------------------

For example, for an EV6 44-bit VA system, 4-way, 256 ASN, 1024 buckets:

 

     using MyITBTraits = PTETraits<CPUFamily::EV6, 44U>;

    using MyDTBTraits = Ev6_DtbPteTraits<CPUFamily::EV6, 44U>;

 

    constexpr unsigned AssocWays   = 4;

    constexpr unsigned MaxASN      = 256;

    constexpr unsigned BucketCount = 1024;

    constexpr bool     ShardBySize = true;

 

    using ITBVictimPolicy = SRRIPPolicy<typename SPAMBucket<MyITBTraits, AssocWays, MaxASN>::Entry, AssocWays>;

    using DTBVictimPolicy = SRRIPPolicy<typename SPAMBucket<MyDTBTraits, AssocWays, MaxASN>::Entry, AssocWays>;

    using ITBInvalidation = DefaultInvalidationStrategy<typename SPAMBucket<MyITBTraits, AssocWays, MaxASN>::Entry>;

    using DTBInvalidation = DefaultInvalidationStrategy<typename SPAMBucket<MyDTBTraits, AssocWays, MaxASN>::Entry>;

 

    using ITBManagerType = SPAMShardManager<MyITBTraits, AssocWays, MaxASN, BucketCount, ShardBySize, ITBVictimPolicy, ITBInvalidation>;

    using DTBManagerType = SPAMShardManager<MyDTBTraits, AssocWays, MaxASN, BucketCount, ShardBySize, DTBVictimPolicy, DTBInvalidation>;

 

Step 2: Instantiate With Correct Parameters (e.g., in SMPManager/SystemManager)

-------------------------------------------------------------------------------

Suppose cpuCount is the number of logical CPUs:

 

    ITBManagerType globalItbManager(cpuCount);

    DTBManagerType globalDtbManager(cpuCount);

 

    // Store these as members of your SMPManager/SystemManager:

    class SMPManager {

        ITBManagerType globalItbManager;

        DTBManagerType globalDtbManager;

        std::vector<AlphaCPU> cpuList;

        // ...

        SMPManager(unsigned cpuCount)

          : globalItbManager(cpuCount), globalDtbManager(cpuCount)

        { /* ... */ }

    };

 

Step 3: Pass to Dependent Classes by Reference or Pointer

---------------------------------------------------------

Each AlphaCPU (or its associated CPUStateIPRInterface) can receive a reference to the global manager, e.g.:

 

    AlphaCPU(ITBManagerType& itbRef, DTBManagerType& dtbRef, /* ... */);

 

And store/use the reference as:

 

    class AlphaCPU {

        ITBManagerType& itbManager;

        DTBManagerType& dtbManager;

        // ...

    };

 

In this model, **the SPAMShardManager is owned by the SMPManager**, not by individual AlphaCPU instances. CPUs cooperate with the central manager for all TLB operations, especially system/global invalidations.

 

Example:

 

 // In a system with 4 CPUs:

    using ITBManagerType = SPAMShardManager<MyITBTraits, 4, 256, 1024, true, ITBVictimPolicy, ITBInvalidation>;

    ITBManagerType globalItbManager(4);

 

    // Use: globalItbManager.tbia_allCPUs();

 

Summary

-------

•The SPAMShardManager (or its equivalent) is best owned by the SMP/system manager.

•SMPManager is responsible for all global invalidation coordination, including IPI delivery and synchronization.

•AlphaCPU objects access and operate on their own shards/slices via references or indices into the central manager.

•This ensures TLB consistency, enables fast SMP invalidation, and supports efficient scalability in multicore systems.
