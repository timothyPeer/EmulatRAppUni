## Notes (Important Architectural Semantics)

 

1. **Per-CPU, Multi-Context HWPCB Model**

• Each CPU owns **multiple HWPCB slots**.

• Slot `cpuId` is the *active execution context*.

• Slots starting at `HWPCB_CONTEXT_SAVE_OFFSET` are **saved contexts** (PAL traps, interrupts, exceptions).

 

This matches **Alpha EV4/EV5/EV6 PALcode behavior**, where:

 

•PALcode executes on a **saved hardware context**

•User context must be restored precisely

 

2. **Snapshot Strategy**

 

Snapshot index is computed as:

 

snapshotSlot = HWPCB_CONTEXT_SAVE_OFFSET + cpuId

 

This guarantees:

•No cross-CPU overwrite

•Deterministic save/restore mapping

•SMP-safe context preservation

 

3. **PAL Mode Synchronization**

 

`forceUserPC()` explicitly:

•* Clears PC bit 0

•* Forces PAL mode flag to false via `globalIPRHot(cpuId)`

 

This correctly models:

* **PC[0] = PAL bit**

* Hardware-enforced PAL ↔ User transitions

 

4. **Global Singleton Pattern**

 

•`GlobalHWPCBController` is a controlled singleton

•Initialization order is explicit

•No accidental static initialization order bugs

 

This is **correct for emulator cores** and avoids:

•

•* TLS misuse

•* Static lifetime hazards

•* Cross-DLL initialization issues
