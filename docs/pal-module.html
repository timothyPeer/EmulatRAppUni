PALcode defines several queue-related concepts:

Interrupt queues

AST queues

Soft interrupt / software trap queues

Machine check logout queues

Instruction restart queues (EV6 only)

Memory barrier / IPI rendezvous queues

Process control block / context queues

But architecturally:

📌 Each CPU has its own private PAL queue structures.

This includes:

✔ Per-CPU Interrupt Queue

The interrupt dispatch sequence is defined in PAL, and the vector logic is per CPU:

Device IRQ → delivered to target CPU

CPU stores it in its PAL temp/queue region

PAL checks and drains that CPU’s queue

✔ Per-CPU AST Queue

ASTs do not cross CPUs; the OS arranges delivery to a specific CPU.

✔ Per-CPU Machine Check Logout Frame Queue

Each CPU maintains its own logout area (in memory) and PAL uses it during trap entry.

✔ Per-CPU IPR queue semantics

Many PAL instructions access per-CPU IPR banks; no IPR queues are shared.

DRIVERSCHED is OS-level, implemented using PAL + IPIs.

This is not a hardware instruction.

This is not a public PALcall.

This is not a documented IPR.

It is a process used by OSF/1 and OpenVMS kernels.

 

🟦 2. What is a Rendezvous Operation?

Rendezvous in Alpha SMP means:

 

“Force all CPUs to enter a known PAL state, wait until all are present, perform a global action, then release them.”

 

This is used for operations that must be globally consistent, such as:

✔ TBIA / TBIAP (global TLB invalidation)

✔ SCB / vector table replacement

✔ ASN epoch rollover

✔ Global memory barriers

✔ Rebalancing AST queues

✔ Reprogramming clock sources

✔ Rescheduling the processor sets

Rendezvous looks like a global lock, but at the PAL level.

The OS essentially says:

 

 

CPU0 broadcasts a special IPI (“enter rendezvous”).

 

 

Every CPU traps into PAL, executing the rendezvous entrypoint.

 

 

CPU0 waits until all CPUs are in rendezvous.

 

 

CPU0 performs the global action (such as TBIA).

 

 

CPU0 signals “exit rendezvous.”

 

 

All CPUs return back to the OS kernel.

 

 

This guarantees global consistency.

 

📌 3. DRIVERSCHED + Rendezvous together

DRIVERSCHED is one of the subtypes of SMP rendezvous operations.

It is used when the OS scheduler (e.g., on OpenVMS) must reconfigure:

 

 

Run queues

 

 

Global scheduling parameters

 

 

Quantum expiration models

 

 

System timebase interactions

 

 

Process set redistribution

 

 

CPU affinity adjustments

 

 

Because scheduling changes involve keeping all CPU run queues coherent, you cannot change scheduling policies or queue structure on one CPU without quiescing the others.

Thus:

✔ DRIVERSCHED calls → trigger a Rendezvous

✔ Rendezvous forces all CPUs into a known state

✔ PAL and OS coordinate the schedule change

✔ CPUs resume normal operation after synchronization

 

📚 4. PAL’s Role in DRIVERSCHED / Rendezvous

The PALcode provides:

A. IPI delivery for rendezvous

A standard “rendezvous IPI vector” is delivered to all CPUs.

B. PAL-level enter/exit handlers

PAL traps the CPU into a known state (mode, IPL, PC shadowing).

C. Register shadowing

PAL shadow registers preserve state while the CPU is quiesced.

D. Synchronization flags

PAL uses per-CPU flags to indicate:

 

 

“I have entered rendezvous”

 

 

“I am ready to be released”

 

 

E. Micro-barriers

PAL ensures no CPU continues until the global action is complete.

 

🧩 5. Does DRIVERSCHED matter for your emulator?

Mostly no, unless:

 

 

You want to emulate VMS or Tru64 kernel-level scheduling behavior, or

 

 

You want to fully support OpenVMS SMP boot.

 

 

But emulator-level consequences are useful:

✔ When an OS broadcasts TBIA, you must use IPI rendezvous

✔ When an OS changes scheduling parameters, CPUs must quiesce

✔ When an OS enters certain critical regions, all CPUs must sync

✔ Rendezvous is the mechanism for global TLB invalidation

✔ PAL TBIS/TBIAP grains cooperate with it

So while DRIVERSCHED itself does not need to be implemented as a separate command, rendezvous semantics absolutely do.

 

🧱 6. For EMulatR, how to represent this?

You don’t need to emulate the actual "DRIVERSCHED" function.

But you must implement:

📌 IPI-based SMP rendezvous API inside IRQController

void sendRendezvousIPI(cpuId);

void broadcastRendezvousIPI();

 

📌 Per-CPU rendezvous flags inside AlphaCPU

std::atomic<bool> inRendezvous;

std::atomic<bool> rendezvousAck;

 

📌 Global synchronization barrier

Inside IRQController:

void waitForAllCPUsInRendezvous();

void releaseRendezvous();

 

📌 Use case: TBIA / TBIAP

This is the most common rendezvous action:

broadcastIPI(REND_IPI);

waitForAllCPUsInRendezvous();

ev6PteCache.invalidateAll();

releaseRendezvous();

 

This is how TBIA/TBIAP really work at OS level.

 

🧠 7. Clear one-liner definition

DRIVERSCHED:

A kernel-level rendezvous operation used by OpenVMS and OSF/1 to synchronize all CPUs while reorganizing scheduling data structures.

Rendezvous:

A PAL-assisted SMP synchronization mechanism ensuring all CPUs enter a safe state before executing global actions like TLB invalidation or scheduling reconfiguration.
