# GuestMemory SRM Integration - Complete Guide

 

## Summary

 

You asked how to add **SRM as a routing target** in your existing GuestMemory architecture. Here's the complete solution:

 

---

 

## 🎯 Your Current Architecture (Excellent!)

 

### **Existing Routing System:**

 

```cpp

GuestMemory (router)

   ↓ classifies PA

   ├─→ SafeMemory (RAM)              0x0 - 0x1_0000_0000 (4GB)

   ├─→ MMIOManager (devices)          0x1_0000_0000 - 0x2_0000_0000 (4GB)

   └─→ AlphaMemorySystem (PAL+SRM)    0x2_0000_0000 - 0x2_0500_0000 (80MB)

```

 

**Your comment says AMS handles "PAL+SRM+HWRPB"** - but these are conceptually different:

- **PAL** = Low-level CPU services (C++ emulated or real Alpha binary)

- **SRM** = High-level firmware (clipper.bin - real Alpha binary)

- **HWRPB** = Data structures

 

**Recommendation:** Separate SRM from AMS for clarity and modularity.

 

---

 

## ✅ Proposed Architecture (4 Routing Targets)

 

### **Updated Routing System:**

 

```cpp

GuestMemory (router)

   ↓ classifies PA

   ├─→ SafeMemory (RAM)              0x0_0000_0000 - 0x1_0000_0000 (4GB)

   ├─→ MMIOManager (devices)         0x1_0000_0000 - 0x2_0000_0000 (4GB)

   ├─→ AlphaMemorySystem (PAL)       0x2_0000_0000 - 0x2_0010_0000 (1MB)

   ├─→ SRMFirmwareRegion (SRM)       0x2_0010_0000 - 0x2_0030_0000 (2MB) ← NEW!

   └─→ AlphaMemorySystem (HWRPB)     0x2_0030_0000 - 0x2_0040_0000 (1MB)

```

 

**Benefits:**

- ✅ Clear separation: PAL ≠ SRM ≠ HWRPB

- ✅ SRM can be reloaded independently

- ✅ Write protection enforced at SRM level

- ✅ Statistics tracking for SRM accesses

- ✅ Easy to understand and maintain

## 🔧 Implementation Changes

 

### **1. Add SRM to RouteTarget Enum**

 

```cpp

enum class RouteTarget : quint8

{

   None = 0,

   SafeMemory,

   MMIOManager,

   AlphaMemorySystem,

   SRMFirmware        // ← NEW!

};

```

 

### **2. Create SRMFirmwareRegion Class**

 

**New class to handle SRM binary:**

- Loads clipper.bin from file

- Provides read/write access (write-protected by default)

- Tracks access statistics

- Enforces alignment and bounds

 

**Files:**

- `SRMFirmwareRegion.h` - Interface

- `SRMFirmwareRegion.cpp` - Implementation

 

### **3. Update GuestMemory**

 

**Add SRM subsystem:**

```cpp

class GuestMemory {

   // ...

   SRMFirmwareRegion* m_srm{ nullptr };  // ← NEW!

   

   void attachSubsystems(

       SafeMemory* sm,

       MMIOManager* mmio,

       AlphaMemorySystem* ams,

       SRMFirmwareRegion* srm  // ← NEW!

   );

};

```

 

**Update routing table:**

```cpp

void GuestMemory::initDefaultPARoutes() noexcept

{

   m_routes.clear();

   

   // RAM: 4GB at 0x0

   m_routes.append({ 0x0000000000000000ULL,

                     0x0000000100000000ULL,

                     RouteTarget::SafeMemory });

   

   // MMIO: 4GB at 0x1_0000_0000

   m_routes.append({ 0x0000000100000000ULL,

                     0x0000000200000000ULL,

                     RouteTarget::MMIOManager });

   

   // AMS PAL: 1MB at 0x2_0000_0000

   m_routes.append({ 0x0000000200000000ULL,

                     0x0000000200100000ULL,

                     RouteTarget::AlphaMemorySystem });

   

   // SRM: 2MB at 0x2_0010_0000 ← NEW!

   m_routes.append({ 0x0000000200100000ULL,

                     0x0000000200300000ULL,

                     RouteTarget::SRMFirmware });

   

   // AMS HWRPB: 1MB at 0x2_0030_0000

   m_routes.append({ 0x0000000200300000ULL,

                     0x0000000200400000ULL,

                     RouteTarget::AlphaMemorySystem });

}

```

 

**Update routing logic:**

```cpp

MEM_STATUS GuestMemory::readRouted(quint64 pa, quint8 width,

                                  quint64& outValue, AccessKind kind) noexcept

{

   const RouteTarget t = classifyPARange(pa, width);

   

   switch (t) {

   case RouteTarget::SafeMemory:

       return m_safeMem->load(pa, width, outValue);

       

   case RouteTarget::MMIOManager:

       return m_mmio->handleRead(pa, width, outValue);

       

   case RouteTarget::AlphaMemorySystem:

       return m_ams->read(pa, width, outValue);

       

   case RouteTarget::SRMFirmware:  // ← NEW!

       {

           const quint64 offset = pa - m_srm->basePA();

           return m_srm->read(offset, width, outValue);

       }

       

   default:

       return MEM_STATUS::AccessViolation;

   }

}

```

## 🚀 Initialization Sequence

 

### **Step 1: Create Subsystems**

 

```cpp

// Create all four subsystems

SafeMemory* safeMem = new SafeMemory();

safeMem->allocate(0x0, 0x100000000);  // 4GB RAM

 

MMIOManager* mmio = new MMIOManager();

 

AlphaMemorySystem* ams = new AlphaMemorySystem();

 

SRMFirmwareRegion* srm = new SRMFirmwareRegion({

   .basePA = 0x0000000200100000ULL,

   .size = 0x00200000ULL,  // 2MB

   .writeProtected = true

});

```

 

### **Step 2: Load SRM Firmware**

 

```cpp

// Load clipper.bin into SRM region

if (!srm->loadFromFile("clipper.bin")) {

   ERROR_LOG("Failed to load SRM firmware");

}

```

 

### **Step 3: Attach to GuestMemory**

 

```cpp

auto& gm = global_GuestMemory();

 

gm.attachSubsystems(

   safeMem,  // RAM

   mmio,     // Devices

   ams,      // PAL/HWRPB

   srm       // SRM firmware ← NEW!

);

 

gm.initDefaultPARoutes();

```

 

### **Step 4: Set CPU PC**

 

```cpp

// Set PC to SRM entry point

cpu->setPC(0x0000000200100000ULL);  // SRM base

 

// Set PAL_BASE (for C++ emulation, this is unused)

cpu->writeIPR(IPR_PAL_BASE, 0x0000000200000000ULL);

```

## 💡 How It Works

 

### **Instruction Fetch from SRM:**

 

```

CPU wants to fetch instruction at PC = 0x2_0010_0000

 

1. CPU calls: gm.readInst32(0x2_0010_0000, instr)

2. GuestMemory classifies PA → RouteTarget::SRMFirmware

3. GuestMemory routes to: m_srm->read(offset=0, width=4, value)

4. SRMFirmwareRegion reads from clipper.bin buffer

5. Returns Alpha instruction (e.g., CALL_PAL #0x03)

6. CPU executes instruction

```

 

### **CALL_PAL Execution:**

 

```

SRM code contains: CALL_PAL #0x03 (CSERVE)

 

1. CPU decodes CALL_PAL instruction (fetched from SRM region)

2. CPU traps to PAL mode

3. Your PalService::executeCSERVE() is invoked (C++ handler)

4. R16 examined for CSERVE function code

5. Console I/O performed via ConsoleManager

6. Return to SRM code (next instruction)

```

 

### **Write Protection:**

 

```

Attempt to write to SRM:

 

gm.write64(0x2_0010_0000, 0x12345678)

 ↓

GuestMemory routes to SRMFirmwareRegion

 ↓

SRMFirmwareRegion checks writeProtected = true

 ↓

Returns MEM_STATUS::AccessViolation

 ↓

CPU raises fault

```

## 🎯 Key Design Decisions

 

### **Why Separate SRM from AMS?**

 

**Conceptual:**

- PAL = Low-level system services (often C++ emulated)

- SRM = High-level firmware (real Alpha binary)

- HWRPB = Data structures

- **These are different things!**

 

**Practical:**

- SRM binary can be reloaded independently

- Write protection enforced at SRM level

- Statistics tracking for SRM execution

- Clearer debugging (know if executing from SRM vs PAL)

- Easier to swap SRM versions

 

### **Why Not Make SRM Part of SafeMemory?**

 

SafeMemory is for **general RAM** - writable, cacheable, no special handling.

 

SRM needs:

- Write protection (ROM)

- Load from file (clipper.bin)

- Statistics tracking

- Separate lifecycle management

 

**Separate class is cleaner.**

 

### **Why 0x2_0010_0000 for SRM Base?**

 

Your existing map uses 0x2_xxxx_xxxx for system resources:

- 0x2_0000_0000 = PAL (1MB is plenty for PAL if you had it)

- 0x2_0010_0000 = SRM (2MB for clipper.bin)

- 0x2_0030_0000 = HWRPB (1MB for data structures)

 

**Total:** 4MB of system space, well within your 0x2_0500_0000 allocation.
