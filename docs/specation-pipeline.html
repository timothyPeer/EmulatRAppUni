ALPHA AXP PIPELINE EXECUTION SPECIFICATION

Authoritative Coding Specification (Pipeline-Centric)

===============================================================================

 

1. PURPOSE AND SCOPE

--------------------

 

This specification defines the authoritative execution model for the Alpha AXP

emulator pipeline.  The pipeline is the single source of truth during instruction

execution.  AlphaCPU serves as the orchestration and control boundary only.

 

This document specifies:

- Required control flow

- Mandatory stage entry/exit behavior

- Stall and fault semantics

- Box responsibilities

- Precise exception and redo guarantees

 

The specification is pipeline-centric and applies to all instruction classes.

Detailed execution is illustrated using MBox (memory) operations, which subsume

the control requirements of EBox, FBox, CBox, and PalBox.

 

-------------------------------------------------------------------------------

 

2. ARCHITECTURAL ROLES

---------------------

 

2.1 AlphaCPU

------------

 

AlphaCPU:

- Owns the run loop

- Owns architectural state (PC, registers, IPRs)

- Owns FaultDispatcher / faultSink

- Enters and exits PAL mode

- Does NOT execute instruction semantics

- Does NOT resolve stalls

- Does NOT advance PC speculatively

 

AlphaCPU SHALL:

- Call IBox::stepPipeline()

- Call Pipeline::step()

- Observe pipeline termination conditions

- Deliver pending faults into PAL

- Resume execution at the PC specified by the architecture

 

2.2 IBox (Frontend)

-------------------

 

IBox:

- Fetches instruction grains

- Performs ITB translation for fetch

- Performs decode cache lookup

- Produces FetchResult objects

- Never commits architectural state

 

IBox SHALL:

- Respect frontend stall indications

- Stop fetching immediately on stall or fault

- Return control to AlphaCPU without side effects

 

2.3 AlphaPipeline (Authoritative Execution Engine)

--------------------------------------------------

 

The pipeline:

- Is the sole authority for instruction progression

- Owns PipelineSlot state

- Owns stall and fault propagation

- Determines whether instructions retire, stall, or fault

 

No other component may override pipeline decisions.

 

-------------------------------------------------------------------------------

 

3. PIPELINE CONTROL INVARIANTS

-----------------------------

 

INV-1: The pipeline is the source of truth for instruction state.

 

INV-2: A faulting instruction MUST NOT retire.

 

INV-3: On synchronous fault, the architectural PC MUST remain at the faulting

      instruction.

 

INV-4: Redo semantics are mandatory for ITB/DTB misses.

 

INV-5: A stall is a microarchitectural condition; a fault is an architectural

      condition.  They MUST NOT be conflated.

 

INV-6: Boxes may detect faults but may not advance PC or retire instructions.

 

-------------------------------------------------------------------------------

 

4. PIPELINE SLOT STATE

---------------------

 

Each PipelineSlot SHALL contain, at minimum:

 

- valid                 : instruction present

- stalled               : microarchitectural stall latch

- faultPending          : architectural fault latch

- execUnit              : EBox / FBox / MBox / CBox / PalBox

- needsWriteback        : indicates architectural result

- payload               : computed result (if any)

- memResultValid        : memory result latch

- stagedITB / stagedDTB : speculative PTE artifacts

 

Rules:

- faultPending dominates stalled

- faultPending MUST be sticky until consumed by AlphaCPU

- stalled MUST block advancement but not raise PAL

 

-------------------------------------------------------------------------------

 

5. FRONTEND STALL SEMANTICS

--------------------------

 

Frontend stall is a derived condition and SHALL be computed as:

 

FrontendStalled :=

   stage(0).valid            OR

   stage(1).stalled          OR

   stage(2).stalled          OR

   stage(3).stalled          OR

   stage(4).stalled          OR

   stage(5).stalled

 

If FrontendStalled is true:

- IBox SHALL NOT fetch

- No new FetchResult SHALL be consumed

- Pipeline SHALL continue resolving internal stalls

 

Frontend stall does NOT:

- Raise faults

- Enter PAL

- Flush pipeline

 

-------------------------------------------------------------------------------

 

6. FAULT SEMANTICS

-----------------

 

Faults are architectural and synchronous.

 

6.1 Fault Detection

-------------------

 

Any box detecting a fault SHALL:

- Set slot.faultPending = true

- Populate faultSink with a PendingEvent

- Return immediately

- Perform NO further semantic work

 

This includes:

- EA calculation failure

- Alignment faults

- ITB/DTB miss

- Access violation

- Memory bus error

 

6.2 Fault Propagation

--------------------

 

Once slot.faultPending is set:

- No further pipeline advancement SHALL occur

- No retirement SHALL occur

- Younger slots SHALL be flushed

- Pipeline::step() SHALL terminate immediately

 

6.3 Fault Consumption

--------------------

 

AlphaCPU SHALL:

- Observe faultPending via pipeline return status

- Enter PAL

- Resume execution at the architectural continuation PC

 

-------------------------------------------------------------------------------

 

7. PIPELINE STEP CONTRACT

-------------------------

 

Pipeline::step() represents ONE PIPELINE CYCLE.

 

Pipeline::step() SHALL:

1. Execute stages in reverse order: WB -> MEM -> EX -> IS -> DE -> IF

2. Abort immediately if faultPending is observed

3. Block advancement if any stage is stalled

4. Advance the ring ONLY if no stall and no fault

 

Pipeline::step() SHALL return control to AlphaCPU after:

- Fault detection

- Stall resolution

- Retirement

- Idle cycle

 

-------------------------------------------------------------------------------

 

8. STAGE SPECIFICATIONS

----------------------

 

8.1 STAGE WB (Writeback / Retirement)

-------------------------------------

 

Responsibilities:

- Detect retirement eligibility

- Commit architectural state

- Advance PC exactly once per retired instruction

 

Rules:

- If slot.faultPending == true:

   - Clear slot

   - Return immediately

- If needsWriteback == true:

   - Commit payload to FP or INT register

- Advance PC ONLY here

- Clear slot.valid after retirement

 

8.2 STAGE MEM (Memory Phase 2)

------------------------------

 

Responsibilities:

- Perform physical memory access

- Finalize load/store results

 

Rules:

- If stalled: hold state

- If faultPending: propagate upward

- Loads set memResultValid

- Stores perform commit or fault

- No PC updates allowed

 

8.3 STAGE EX (Execution)

------------------------

 

Responsibilities:

- Dispatch to execution box

- Perform semantic execution

 

For MBox:

- Calculate effective address

- Perform DTB translation

- Stage PTEs

- Detect faults

 

Rules:

- On fault: set faultPending and return

- On stall (e.g., resource busy): set stalled

 

8.4 STAGE IS (Issue)

--------------------

 

Responsibilities:

- Detect register hazards

- Perform scoreboard updates

 

Rules:

- On hazard: set stalled

- Do not advance on stall

 

8.5 STAGE DE (Decode)

---------------------

 

Responsibilities:

- Decode instruction format

- Extract operands

- Determine execUnit

 

Rules:

- No architectural side effects

- Decode MUST precede execution

 

8.6 STAGE IF (Fetch Injection)

------------------------------

 

Responsibilities:

- Consume FetchResult

- Populate slot(0)

 

Rules:

- If frontend stalled: do nothing

- If faultPending: do nothing

- Advance ring ONLY if slot accepted

 

-------------------------------------------------------------------------------

 

9. MBOX EXECUTION (REFERENCE MODEL)

----------------------------------

 

MBox is the reference execution model.

 

MBox SHALL:

- Read operand values from AlphaProcessorContext

- Never treat register indices as values

- Use staged DTB PTEs if present

- Perform SPAM lookup if not staged

- Set slot.stalled on retryable conditions

- Set slot.faultPending on faults

- Never advance PC

 

All memory faults MUST be precise and restartable.

 

-------------------------------------------------------------------------------

 

10. PAL AND CONTROL FLOW

-----------------------

 

- PAL entry is initiated ONLY by AlphaCPU

- Pipeline MUST quiesce before PAL

- Continuation PC MUST point to faulting instruction

- On PAL return, pipeline restarts at that PC

 

-------------------------------------------------------------------------------

 

11. REQUIRED PIPELINE CHANGES (ASSESSMENT)

------------------------------------------

 

Based on current implementation, the following changes are REQUIRED:

 

- Pipeline::step() must return immediately on faultPending

- advanceRing() must be conditional

- PC advancement must exist ONLY in WB

- faultPending must be set by all faulting boxes

- stalled must block advancement but not raise PAL

- Frontend stall must be consumed by AlphaCPU/IBox

- MBox must read register values, not indices

 

-------------------------------------------------------------------------------

 

12. SUMMARY

-----------

 

This specification establishes a strict, pipeline-authoritative execution model.

All boxes operate as semantic engines only.  All control decisions flow through

the pipeline and terminate in the AlphaCPU run loop.

 

MBox execution fully exercises this model; correctness here implies correctness

for all other boxes.
