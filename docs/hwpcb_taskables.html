# ✅ **Which HWPCB registers correspond to AST?**

 

In the Alpha architecture (EV4/EV5/EV6), **AST handling uses *four distinct registers***:

 

| Register  | Mnemonic           | Purpose                                                            |

| --------- | ------------------ | ------------------------------------------------------------------ |

| **ASTEN** | AST Enable         | Bitmask of which AST levels are *enabled* (allowed to fire).       |

| **ASTER** | AST Enable Request | Holds the enable bits as maintained by PALcode / OS.               |

| **ASTRR** | AST Request        | A write by the OS or PAL requesting an AST at a given level (0–3). |

| **ASTSR** | AST Summary        | Reflects *pending and enabled* ASTs. Hardware-computed.            |

 

✔ **So yes** — ASTER, ASTEN, ASTRR, and ASTSR are all part of the **process context (HWPCB)** and must be saved/restored on a context switch.

 

💡 **AST is part of the “software interrupt” subsystem**, and those fields live exactly where you think they do: the process/HWPCB state.

 

This matches OpenVMS behavior and all EV6 documentation.
