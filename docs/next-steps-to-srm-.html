# Concrete Boot Sequence to SRM

 

*(Alpha AXP Emulator – Operational Checklist)*

 

This is the **minimum correct sequence** required to reach an SRM `>>>` prompt.

 

---

 

## Phase 0 — Static Construction (Before Any CPU Runs)

 

**Goal:** All global subsystems exist, but nothing is “live”.

 

**Must be true:**

 

1. **GuestMemory**

 

  * Physical RAM allocated

  * MMIO regions registered

  * Reservation system initialized (LL/SC empty)

 

2. **MMIO + Devices**

 

  * ConsoleOPA0Device constructed

  * ConsoleManager constructed

  * SRMEnvStore constructed (even if empty)

 

3. **Interrupt Infrastructure**

 

  * IRQController constructed

  * IPIManager constructed

  * IPIMailboxes allocated per CPU

 

4. **PAL Infrastructure**

 

  * PalVectorTable constructed

  * PalVectorTable::initialize() called

  * All PAL vectors registered

 

5. **CPU Objects**

 

  * AlphaCPU objects constructed

  * CPURegistry populated

  * CPU count fixed (SMP known)

 

🚫 **No CPU executes yet**

 

---

 

## Phase 1 — Reset Entry (Power-On State)

 

**Goal:** CPU is in architecturally correct reset state.

 

**On CPU 0 only:**

 

1. **Processor State**

 

  * PC = RESET_VECTOR (architectural constant)

  * Mode = PAL

  * IPL = max (interrupts masked)

  * PS initialized to reset value

  * All integer & FP registers undefined/cleared (implementation choice)

 

2. **Pipeline**

 

  * Completely empty

  * No speculative state

  * No reservations

 

3. **TLB**

 

  * ITB empty

  * DTB empty

 

4. **LL/SC**

 

  * All reservations cleared

 

This corresponds to **hardware reset**, not a PAL call.

 

---

 

## Phase 2 — Initial PAL Execution (RESET PAL Vector)

 

**Goal:** Enter PAL reset handler and bootstrap minimal machine state.

 

**Mechanism:**

 

* You **do not fetch an instruction** yet.

* You **directly vector into PAL**.

 

**Required behavior:**

 

```cpp

PalService::enterPALVector(

   slot,

   PalVectorId::RESET,

   reset_pc,

   empty_args

);

```

 

**PAL RESET handler must:**

 

1. Establish:

 

  * Initial stack pointer

  * HWPCB

  * Exception save area

 

2. Initialize critical IPRs:

 

  * ASN

  * VA control

  * ICCSR

  * PS shadow registers

 

3. Leave CPU in PAL mode

 

📌 At this stage:

 

* No SRM yet

* No console output yet

* Just PAL groundwork

 

---

 

## Phase 3 — Transition from PAL to SRM Firmware Code

 

**Goal:** Begin executing SRM firmware as *normal code*, not PAL.

 

**Critical architectural rule:**

 

> **HW_REI is the ONLY legal exit from PAL mode**

 

Your statement here is correct and essential.

 

**PAL RESET must eventually:**

 

* Set up a return PC pointing into SRM image

* Execute `HW_REI`

 

**HW_REI must restore:**

 

* PC

* Processor mode (to kernel)

* PS

* IPL

* Interrupt enable state

 

**After HW_REI:**

 

* Pipeline resumes normal fetch/decode

* CPU is no longer in PAL mode

* Execution begins in SRM code

 

---

 

## Phase 4 — SRM Firmware Execution Begins

 

**Goal:** SRM runs as ordinary privileged software.

 

**SRM expects:**

 

1. **Instruction Fetch**

 

  * From physical memory (or mapped ROM)

  * ITB translation operational

 

2. **Console Path**

 

  * SRM uses CALL_PAL → CSERVE

  * CSERVE routes to ConsoleManager

  * ConsoleManager routes to OPA0 device

 

This is where your existing pieces **finally connect**.

 

---

 

## Phase 5 — CSERVE Path (Console I/O)

 

**Goal:** SRM can print characters and read input.

 

### Required call path:

 

```

SRM code

 → CALL_PAL CSERVE

   → PalService::executeCSERVE

     → ConsoleManager

       → ConsoleOPA0Device

         → host stdout / socket / terminal

```

 

**Minimum CSERVE functions needed to reach prompt:**

 

* PUTC

* PUTS

* GETC (later)

 

SRM banner output **proves** this path is working.

 

---

 

## Phase 6 — Interrupt Enable & Idle Loop

 

**Goal:** SRM reaches interactive prompt.

 

SRM will:

 

1. Enable interrupts

2. Probe devices

3. Print banner

4. Enter command loop

 

At this point you should see:

 

```

>>>

```

 

🎉 **Boot successful**

 

---

 

# Failure Modes to Watch For (Very Important)

 

If SRM does **not** appear, check these in order:

 

### 1. Never leaving PAL

 

* HW_REI not implemented correctly

* Mode bit not restored

 

### 2. CALL_PAL executes but CSERVE silent

 

* ConsoleManager not initialized

* OPA0 not registered

* CSERVE dispatch table incomplete

 

### 3. Pipeline keeps stalling

 

* Barrier not released

* EXCB/TRAPB incorrectly blocking fetch

* FaultDispatcher never drained

 

### 4. Immediate fault loop

 

* ITB miss not handled

* Translation faults not vectored correctly

* FaultDispatcher priority inversion
