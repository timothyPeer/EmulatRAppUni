Lifecycle

A characterization aligned with EV6/ASA semantics and the current EmulatR design.

 

## Short Answer (Verdict)

 

1. **HW_MTPR (including MTPR_SIRR)** is responsible for **updating IPR state only**.

2. **Interrupts (including ASTs)** are **scheduled**, not immediately executed.

3. **Delivery occurs at a defined post-retirement boundary**, before the next PC+n fetch.

4. **IRQController + FaultDispatcher** are the correct abstraction for delivery.

5. Your current EmulatR model matches Alpha architectural intent.

 

You are **not missing a hidden side effect** in HW_MTPR.

 

---

 

## 1. Architectural Rule to Anchor Everything

 

From the Alpha System Architecture (EV6 consistent):

 

> *Writing an IPR may cause an interrupt or exception to become pending, but delivery occurs only at architecturally defined recognition points.*

 

That sentence is the key.

 

**Recognition ≠ write**

**Pending ≠ delivered**

 

---

 

## 2. Correct Lifecycle (Refined)

 

Let’s restate the lifecycle using your components and the Alpha rules.

 

A. Grain executes (e.g., HW_MTPR_SIRR)

 

At grain execution:

 

* The grain:

 

 * Validates privilege

 * Writes the architectural register (IPR)

 * Returns normally

 

For **MTPR_SIRR** specifically:

 

* Bitwise effect:

 

 * SIRR bit(s) set in the appropriate IPR

* Semantic effect:

 

 * “An interrupt is now *pending*”

* **No control transfer**

* **No PAL entry**

* **No pipeline break**

 

This is *exactly correct*.

 

> HW_MTPR sets state; it does not deliver events.

 

---

 

B. Grain retirement (serialization point)

 

At retirement:

 

* All side effects of the grain are complete

* Ordering is guaranteed

* PC+n is determined but **not yet fetched**

 

This is the **earliest legal point** at which an interrupt may be recognized.

 

---

 

C. Interrupt recognition window (between grains)

 

Now — and only now — the emulator may ask:

 

* Is there a pending interrupt?

* Is delivery enabled?

* Is IPL sufficient?

* Are we not already in PALmode?

 

This check belongs **outside the grain**, in:

 

* AlphaPipeline

* or a post-retirement hook

* or your existing “end of cycle” logic

 

If eligible:

 

* Schedule interrupt delivery via:

 

 * IRQController

 * FaultDispatcher

 

This matches hardware: **interrupt delivery is asynchronous to instruction execution but synchronous to retirement boundaries**.

 

---

 

D. Interrupt delivery (before next fetch)

 

Before fetching PC+n:

 

1. Pipeline drains (conceptually)

2. EXC_ADDR set

3. EXC_SUM updated

4. PALmode entered

5. PC redirected to PAL vector

 

This is handled by **FaultDispatcher**, not the grain that caused the interrupt to become pending.

 

---

 

## 3. Why HW_MTPR Must NOT Deliver Side Effects

 

This is an important architectural correctness point.

 

If HW_MTPR did delivery directly, you would violate:

 

* Precise exception rules

* Interrupt deferral semantics

* PAL nesting guarantees

 

Examples of what must NOT happen inside HW_MTPR:

 

* Jumping to PAL immediately

* Forcing pipeline drain mid-grain

* Overriding PC before retirement

 

Alpha explicitly allows:

 

```

MTPR SIRR

(next instruction retires)

<interrupt is taken here>

```

 

So your separation is **required**, not optional.

 

---

 

## 4. ASTs Are the Same (Just Lower Priority)

 

ASTs (Asynchronous System Traps):

 

* Are architecturally just another interrupt class

* Differ in priority and masking rules

* Use the same delivery mechanism

 

So:

 

* AST pending bit set (via IPR or OS-specific path)

* AST scheduled

* AST delivered at post-retirement recognition point

 

Your model treating ASTs via IRQController + FaultDispatcher is **correct**.

 

---

 

## 5. Mapping Responsibilities (Clear Ownership)

 

This is how you should document ownership:

 

HW_MTPR Grain

 

* Writes IPR

* Updates pending bits

* No delivery

* No control transfer

 

IRQController

 

* Aggregates pending interrupt sources

* Applies priority and masking rules

* Answers “is an interrupt deliverable now?”

 

FaultDispatcher

 

* Performs architectural delivery

* Updates EXC_ADDR / EXC_SUM

* Enters PALmode

* Redirects PC

 

AlphaPipeline

 

* Defines recognition points

* Calls dispatcher before next PC fetch

* Guarantees ordering

 

This is **clean**, **testable**, and **faithful**.

## 7. One-Sentence Documentation You Can Use

 

> *In EmulatR, instructions such as MTPR_SIRR update interrupt-related IPR state but do not directly cause interrupt delivery. Pending interrupts are evaluated only at instruction retirement boundaries by the AlphaPipeline, and delivery is performed by the IRQController and FaultDispatcher prior to fetching the next PC.*

 

That sentence is **architecturally bulletproof**.
