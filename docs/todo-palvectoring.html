================================================================================

Question:

What is needed yet to begin planning bootstrap to SRM?

================================================================================

 

Short answer:

You are *past* the PAL/CPU correctness gate. You are now at the point where

bootstrap planning can begin, but three concrete layers must be completed

or explicitly stubbed before SRM can execute meaningfully.

 

This is not speculative work anymore; it is dependency-driven.

 

================================================================================

BOOTSTRAP DEFINITION (clarity first)

================================================================================

 

For SRM, “bootstrap” means:

 

1) CPU resets into PAL RESET vector

2) PAL establishes a minimal execution environment

3) PAL transfers control to SRM firmware image

4) SRM initializes console devices, memory map, and I/O

5) SRM reaches a visible prompt (>>>)

 

You do NOT need:

- OpenVMS

- Filesystems

- Disks

- Networking

 

You DO need:

- Correct PAL behavior

- Deterministic memory mapping

- Minimal console I/O

 

================================================================================

LAYER 1 — PAL + CPU (STATUS: essentially ready)

================================================================================

 

You already have or are finishing:

 

✔ PAL entry/exit centralized (PalService)

✔ PAL IRQ suppression

✔ Correct IPL SSOT via IRQController

✔ HW_REI restore via snapshot

✔ PAL vector table infrastructure

✔ PC[0] semantics controlled

✔ RESET / MCHK / INTERRUPT vectors defined

 

Remaining *blocking* PAL items (small but mandatory):

 

TODO 1.1) RESET vector must be fully wired

- RESET must:

 - Clear PAL mode state

 - Initialize HWPCB for CPU 0

 - Set PAL_BASE

 - Jump to PAL RESET handler

 

SRM always enters via RESET.

 

TODO 1.2) PAL must provide minimal IPR read/write

SRM will immediately use:

- MFPR / MTPR for:

 - IPL

 - PAL_BASE

 - WHAMI

 - ICCSR

 - TB control registers (even if stubbed)

 

You do not need full semantics yet.

You *must not fault* on access.

 

TODO 1.3) ITB mapping for PAL + SRM image

- PALcode and SRM must execute without I-stream faults.

- Simplest acceptable solution:

 - Identity-map a low physical region permanently

 - Disable ITB faults during PAL/SRM bootstrap

 

This is explicitly allowed by ASA.

 

================================================================================

LAYER 2 — Memory Model (STATUS: partially ready, must finish)

================================================================================

 

SRM assumes a very specific early-memory environment.

 

Mandatory:

 

TODO 2.1) Physical memory must be contiguous and readable

- SRM probes memory early.

- You can lie (fixed-size RAM), but reads must succeed.

 

TODO 2.2) VA -> PA translation model must be stable

Acceptable bootstrap models:

- Flat identity map (VA == PA)

- Or: MMU disabled until SRM enables it

 

Do NOT attempt partial page table correctness yet.

SRM expects to manage this itself.

 

TODO 2.3) No speculative memory faults

- All instruction fetches and data reads during SRM must succeed

- Return zero-filled memory if unimplemented

 

================================================================================

LAYER 3 — Console I/O (STATUS: NOT READY, REQUIRED)

================================================================================

 

This is the largest missing functional piece.

 

SRM requires:

- One writable output device

- One readable input device (even blocking/polling)

 

Minimum viable console:

 

TODO 3.1) Emulate *one* SRM console device

Acceptable targets:

- Serial TTA0

- Graphics not required

- Keyboard optional (polling OK)

 

Implementation:

- Memory-mapped device or port-based

- SRM will probe known addresses

 

TODO 3.2) Implement character putc/getc semantics

SRM prints extremely early:

- Banner

- CPU type

- Memory size

- “>>>”

 

If putc does not work, you will think SRM is dead when it is not.

 

================================================================================

LAYER 4 — Platform Model (STATUS: can be stubbed)

================================================================================

 

SRM expects some platform identity.

 

TODO 4.1) WHAMI and CPU type

- Return a known Alpha system (ES40 is fine)

- Hardcode if necessary

 

TODO 4.2) Minimal PCI / I/O probing responses

- SRM probes buses

- You may:

 - Return “no device”

 - Or stub a minimal PCI root

 

SRM tolerates missing devices surprisingly well.

 

================================================================================

LAYER 5 — Firmware Image Handling (STATUS: planning phase)

================================================================================

 

You must decide how SRM is loaded and invoked.

 

TODO 5.1) Decide SRM image format

Typical options:

- Raw .SYS image loaded at fixed PA

- Extracted SRM ROM blob

 

TODO 5.2) Decide PAL -> SRM handoff mechanism

Common model:

- PAL RESET vector jumps to SRM entry point

- PC set to SRM start address

- PAL mode exited or partially retained (implementation-specific)

 

TODO 5.3) Instrument first execution

You *must* have:

- Instruction trace

- Console output trace

- Early breakpoint/log when SRM entry is reached

 

================================================================================

WHAT YOU DO NOT NEED YET (important)

================================================================================

 

You do NOT need:

- Disk controllers

- Filesystems

- Networking

- SCSI

- Full MMU correctness

- Full interrupt routing

 

SRM will happily reach >>> without them.

 

================================================================================

ORDERED NEXT STEPS (ACTIONABLE)

================================================================================

 

STEP 1) Finalize PAL RESET path

- Make RESET vector deterministic

- Jump to SRM entry reliably

 

STEP 2) Lock down memory model

- Identity map

- No I-stream faults

- Zero-fill on read

 

STEP 3) Implement minimal console output

- Even printf-to-log is acceptable initially

 

STEP 4) Load SRM image and jump

- Instrument heavily

- Expect early failures

 

STEP 5) Iterate only where SRM touches next

- Do not pre-implement features SRM does not yet reach

 

================================================================================

REALITY CHECK

================================================================================

 

You are no longer blocked by:

- PAL architecture

- CPU correctness

- IRQ/REI semantics

 

You are blocked by:

- Console I/O

- Early memory assumptions

- RESET-to-SRM handoff mechanics

 

That is exactly where a real Alpha platform spends most of its firmware effort.
