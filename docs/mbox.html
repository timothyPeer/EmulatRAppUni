MBox

 

Below is a **clean, grain-scoped specification** for **MBox responsibilities**, deliberately restricted to **what instruction grains executed in the MBox must support** — not pipeline orchestration, not global policy, not device behavior.

 

This is written as a **design contract** you can keep next to the MBox executor header.

 

---

 

# MBox Grain Support Specification

 

(Alpha AXP – EV6-style model)

 

## 1. Scope and Intent

 

The **MBox** is responsible for executing **memory-affecting and ordering-affecting instruction grains** whose architectural effects include:

 

* Data memory access

* Atomicity / reservation semantics

* Memory ordering guarantees

* Cache / prefetch / hint semantics

* Certain PAL-mediated memory operations

 

This specification defines **only what an individual MBox grain must implement** to be architecturally correct.

 

The MBox **does not**:

 

* Own register writeback

* Resolve PAL vectors

* Perform instruction fetch or decode

* Implement scheduling or hazard detection

* Implement device behavior

 

---

 

## 2. Instruction Classes Assigned to MBox

 

An instruction grain is executed in the MBox **iff** it has at least one of the following semantics:

 

* `S_MemFmt`

* `S_Load` or `S_Store`

* `S_LoadLocked` or `S_StoreConditional`

* `S_Barrier`

* Explicit cache / prefetch / write-hint semantics

* PAL instructions whose architectural effect is memory-side

 

2.1 Memory-Format Instructions

 

Examples:

 

* `LDQ`, `LDQ_U`, `LDT`, `LDS`, `LDG`, `LDF`

* `STQ`, `STQ_U`, `STT`, `STS`, `STG`, `STF`

 

2.2 Atomic / Reservation Instructions

 

Examples:

 

* `LDL_L`, `LDQ_L`

* `STL_C`, `STQ_C`

 

2.3 Memory Ordering / Barriers

 

Examples:

 

* `MB`, `WMB`, `TRAPB`

 

2.4 Cache / Hint Instructions

 

Examples:

 

* `FETCH`, `FETCH_M`

* `WH64`, `WH64EN`

* `ECB`, `EXCB`

 

2.5 PALcode Memory Operations

 

Examples:

 

* `PAL_LDQP`

* `PAL_STQP`

* `PAL_CFLUSH`

* `PAL_DRAINA`

 

---

 

## 3. Mandatory Responsibilities of an MBox Grain

 

Each MBox grain **must implement all applicable items below**.

 

---

 

## 4. Effective Address Formation

 

**Every MBox grain that accesses memory must:**

 

1. Compute VA as:

 

  ```

  VA = Rb + signext(disp16)

  ```

 

2. Apply instruction-specific address modification:

 

  * `_U` instructions: clear low bits or compute lane index

  * Alignment-sensitive instructions: validate natural alignment

 

3. Record `faultVA` on success or failure

 

---

 

## 5. Alignment Handling

 

Each grain must enforce alignment rules **exactly per instruction**:

 

| Instruction           | Alignment Required     |

| --------------------- | ---------------------- |

| LDQ / STQ             | 8 bytes                |

| LDT / STT             | 8 bytes                |

| LDS / LDF / STS / STF | 4 bytes                |

| LDQ_U / STQ_U         | **No alignment fault** |

 

If alignment fails:

 

* The grain must construct a **PendingEvent** with:

 

 * `MemoryFaultType::ALIGNMENT`

 * `faultVA = VA`

 * `isWrite` correctly set

* Execution must stop immediately

 

---

 

## 6. Address Translation and Access Control

 

Each MBox grain must:

 

1. Perform **DTB lookup** using:

 

  * `VA`

  * current `ASN`

  * data realm

2. On failure:

 

  * Raise DTB miss PendingEvent

3. Validate access permissions:

 

  * Read vs write

  * Kernel vs user

4. On violation:

 

  * Raise access-violation PendingEvent

 

The grain **must not**:

 

* Modify TLBs

* Perform page table walks directly

* Resolve PAL vectors

 

---

 

## 7. Physical Memory Access

 

Each MBox grain must:

 

* Translate `VA → PA`

* Perform physical memory access using SafeMemory:

 

 * Byte / long / quad as required

* Detect and propagate:

 

 * Bus errors

 * Device faults

 * MMIO exceptions

 

On failure:

 

* Raise a memory fault PendingEvent

* Preserve `faultVA`

 

---

 

## 8. Reservation Semantics (LL/SC)

 

For grains with `S_LoadLocked` or `S_StoreConditional`:

 

Load-Locked (`LD*_L`)

 

* Establish reservation:

 

 * `ReservationManager::setReservation(cpuId, PA)`

* Load value normally

 

Store-Conditional (`ST*_C`)

 

* Verify reservation:

 

 * On success:

 

   * Perform store

   * Return success (payload = 1)

 * On failure:

 

   * Do **not** store

   * Return failure (payload = 0)

* Clear reservation in all cases

 

MBox grains **must** call reservation invalidation on:

 

* Any successful store

* Any conflicting store to the same PA

 

---

 

## 9. Payload Production

 

Each MBox grain must produce exactly one of:

 

* **No payload** (stores, barriers, cache ops)

* **One 64-bit payload** (loads, LL/SC result)

 

Rules:

 

* Payload is an **opaque architectural image**

* FP loads place mapped FP bits into payload

* Integer loads place integer bits into payload

* Store-conditional writes success/failure indicator

 

Set:

 

```

slot.payload

slot.needsWriteback

```

 

The grain **must not** write registers directly.

 

---

 

## 10. Memory Ordering and Barriers

 

For instructions with `S_Barrier`:

 

* The grain must:

 

 * Enforce ordering with respect to:

 

   * Prior loads

   * Prior stores

* Implementation may be:

 

 * Logical fence

 * Pipeline serialization

 * Reservation flush

* Exact mechanism is implementation-defined

 

The grain must **not**:

 

* Perform any memory access

* Modify registers

 

---

 

## 11. Cache / Hint Instructions

 

For cache or hint instructions:

 

* Grains must:

 

 * Compute VA

 * Perform translation and permission checks if required

 * Notify cache / memory subsystem

* Architectural state **must not change**

* No payload is produced

 

---

 

## 12. PAL Memory Grains (MBox side)

 

PAL grains executed in MBox must:

 

* Obey all rules above

* Use physical addressing if specified

* Bypass ASN where architecturally required

* Defer PAL vector resolution to the dispatcher

 

---

 

## 13. Forbidden Responsibilities (Explicitly Out of Scope)

 

An MBox grain must **never**:

 

* Write integer or FP registers directly

* Update PC

* Modify PS or IPL

* Resolve PAL vectors

* Perform instruction decode

* Access other CPUs’ registers

* Manage pipeline scheduling

 

---

 

## 14. Minimal Required Interfaces

 

An MBox grain assumes availability of:

 

* `globalEv6SPAM()` (DTB access)

* `globalFaultDispatcher(cpuId)`

* `SafeMemory` PA read/write

* `ReservationManager`

* `PipelineSlot` (payload + bookkeeping)

 

---

 

## 15. Summary (One-Page Mental Model)

 

> **An MBox grain is responsible for everything that happens *between* a virtual address and an architectural payload — and nothing beyond that.**

 

It:

 

* Computes addresses

* Enforces alignment

* Performs translation

* Applies permissions

* Touches memory

* Manages reservations

* Produces payloads

* Raises faults

 

It does **not**:

 

* Write registers

* Interpret results

* Control execution flow
