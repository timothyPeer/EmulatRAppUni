Endianness Rules

Alpha is ALWAYS little-endian. There is no big-endian mode. Any big-endian data must be explicitly byte-swapped by software.

 

 

ALPHA ENDIANNESS - ARCHITECTURAL FACTS:

---------------------------------------

 

1.ALPHA IS LITTLE-ENDIAN (Native)

a.   - All Alpha processors (21064, 21164, 21264/EV6) are little-endian

b.   - Memory is stored with least significant byte at lowest address

c.   - This is the ONLY native mode - no big-endian mode switch

 

2.INSTRUCTIONS ARE LITTLE-ENDIAN

a.   - 32-bit instructions stored in little-endian format

b.   - Opcode extraction works on little-endian values

c.   - No byte swapping needed for instruction fetch

 

3.DATA ACCESSES ARE LITTLE-ENDIAN

a.   - LDQ, LDL, LDWU, LDBU all assume little-endian memory

b.   - STQ, STL, STW, STB all write little-endian

c.   - No automatic byte swapping in hardware

 

4.BYTE SWAPPING IS SOFTWARE-CONTROLLED

a.   - Alpha provides INSWL, EXTQH, etc. for manual byte manipulation

b.   - OSF/1, Tru64, Linux/Alpha all assume little-endian

c.   - Big-endian data must be explicitly converted by software

 

5.MMIO CAN BE CONFIGURED

a.   - Some Alpha systems (like AlphaServer) have I/O controllers

b.     that support per-device endianness configuration

c.   - This is handled by the I/O bridge/chipset, NOT the CPU

d.   - The CPU always operates in little-endian mode

 

Architecture Stack

--------------------------------------------

 

1.LAYER 1: CPU (Alpha Instructions)

a. Native little-endian operations

b. No byte swapping

c. Operates on host-endian values in registers

 

2.LAYER 2: GuestMemory / SafeMemory

a.Stores memory in LITTLE-ENDIAN format (Alpha native)

b.readPA() converts from little-endian to host-endian

c.writePA() converts from host-endian to little-endian

d.This is WHERE endianness conversion happens

 

3.LAYER 3: MMIO Manager

a.Per-device endianness configuration

b.Most devices: little-endian (native Alpha)

c.Some devices: big-endian (e.g., network interfaces)

d.Handled by device-specific read/write handlers

 

4.LAYER 4: Host Machine

a.Could be x86-64 (little-endian) or ARM64 (bi-endian)

b.Qt provides qFromLittleEndian() / qToLittleEndian()

c.Use these for memory <-> register transfers

MMIO ENDIANNESS HANDLING.

 

MMIO devices may need different endianness:

 

Network Interface Card (Big-Endian)

--------------------------------------------

class NetworkDevice : public MMIODevice

{

    MMIOStatus handleRead(quint64 offset, quint8 width, quint64& outValue) override

    {

        quint32 rawValue = m_registers[offset / 4];

        

        // This device is big-endian (common for network chips)

        if (width == 4) {

            outValue = qFromBigEndian(rawValue);

        }

        

        return MMIOStatus::OK;

    }

    

    MMIOStatus handleWrite(quint64 offset, quint8 width, quint64 value) override

    {

        // Convert from host to big-endian for this device

        if (width == 4) {

            m_registers[offset / 4] = qToBigEndian(static_cast<quint32>(value));

        }

        

        return MMIOStatus::OK;

    }

};

 

Example: Serial Port (Little-Endian)

------------------------------------

class SerialDevice : public MMIODevice

{

    MMIOStatus handleRead(quint64 offset, quint8 width, quint64& outValue) override

    {

        // Most serial devices are byte-oriented, no endianness issues

        if (width == 1) {

            outValue = m_registers[offset];

        }

        

        return MMIOStatus::OK;

    }

};

 

Implementation Rule:

---------

•RAM is always little-endian (Alpha native)

•MMIO endianness is device-specific

•Handle it in the device's read/write methods

•NOT in the CPU instruction handlers

 

INSTRUCTION FETCH ENDIANNESS

 

Instructions are 32-bit little-endian values:

 

Example instruction: 0x47E03400 (little-endian in memory)

Memory bytes: [0x00, 0x34, 0xE0, 0x47] at addresses [n, n+1, n+2, n+3]

 

When fetched:

   quint32 raw;

   readPA_Long(pc, raw);  // Returns 0x47E03400 in host endianness

 

Decode:

   opcode = (raw >> 26) & 0x3F;  // Bits 31:26

   ra = (raw >> 21) & 0x1F;      // Bits 25:21

   

Implementation Rule:

1. Memory stores instruction in little-endian

2. readPA_Long() converts to host endianness

3. Bit extraction works on host-endian value

Same as x86-64 case - qFromLittleEndian() is a no-op ✅

WHERE ENDIANNESS IS HANDLED:

 

•GuestMemory/SafeMemory Layer:

  - Stores bytes in little-endian (Alpha native)

  - Converts to/from host endianness on read/write

  - Uses qFromLittleEndian() / qToLittleEndian()

 

•MMIO Layer (Device-Specific):

  - Per-device endianness configuration

  - Most devices: little-endian (Alpha native)

  - Some devices: big-endian (handled in device code)

 

❌ CPU Instruction Layer (MBox, EBox, etc.):

  - NO endianness handling needed

  - Operates on host-endian values

  - Memory layer handles conversion

 

❌ Register File:

  - NO endianness handling needed

  - Just stores native host values

assert(bytes[1] == 0xCD);

assert(bytes[4] == 0x67);

1. Write converts host → little-endian

2. Memory stores in little-endian

3. Read converts little-endian → host
