1. Purpose

 

This specification defines how Alpha integer register R31 is consumed, produced, and modeled in the AlphaPipeline (including stage_WB), with special  attention to:

 

* R31 as a source operand (RAZ semantics: reads as zero)

* R31 as a destination operand (writes discarded)

* Corner cases where use of R31 as a destination produces UNPREDICTABLE  architectural state (notably LDx_L / STx_C)

* Corner cases where R31 is required in Ra or Rb fields (CTLZ/CTPOP/CTTZ,  SEXTx, AMASK, IMPLVER, ITOFx, etc.)

* Implications for trap-shadow qualification and exception signaling

 

Primary reference: Alpha Architecture rules as summarized in your notes, including Section 4.7.3 ("true zero"), Appendix A timing/atomic notes, and the operand constraints described in the ISA text.

 

Scope: Integer register R31. (F31 rules are similar but more severe and are not normative for this document except where noted for contrast.)

 

2. Definitions and Terms

 

R31:

The architected integer register with special behavior:

- As a source operand: supplies integer zero (0).

- As a destination operand: discards the computed result (no register write).

 

RAZ:

"Reads As Zero" - any read of R31 returns 0.

 

WB:

Writeback stage in AlphaPipeline that commits register and other architectural results.

 

UNPREDICTABLE:

Architecturally permitted behavior where the hardware may produce arbitrary

results or state, and software must not rely on any particular outcome.

 

NOTE: "UNPREDICTABLE" is NOT "UNDEFINED" in the C sense; it means software may

not assume any defined outcome, but implementations may choose a deterministic

behavior for practicality. Your emulator must decide and document a policy.

3. Architectural Rules for R31

 

3.1 R31 as a Source Operand (RA or RB)

 

Rule R31-SRC-1:

If an instruction reads R31 as an operand (Ra or Rb), the operand value is 0.

 

Implementation requirement:

 

* readIntReg(31) must always return 0 regardless of any internal storage.

* No scoreboard dependency should ever be introduced for R31 reads.

 

3.2 R31 as a Destination Operand (Rc / Ra in some formats)

 

Rule R31-DST-1:

If an instruction specifies R31 as a destination integer register, the

computed integer result is discarded (no architectural register update).

 

Implementation requirement:

 

* stage_WB must suppress writes to integer register 31.

* The instruction is otherwise executed normally UNLESS the ISA explicitly

 states reduced execution or UNPREDICTABLE side effects.

 

3.3 "May Not Execute" Note for R31/F31 as Destination

 

Your notes include:

"Because Alpha implementations need not execute instructions that have R31 or

F31 as the destination operand, instructions with such a destination should

not be thought to end a trap shadow."

 

Rule R31-DST-2 (Trap shadow qualification):

An instruction with destination R31 must not be used by the OS/compiler as a

reliable trap-shadow terminator.

 

Emulator policy:

For correctness, your emulator SHOULD still execute the instruction and

preserve normal exception behavior, except for cases explicitly stated as

UNPREDICTABLE or prefetch-hint forms.

 

Practical outcome:

 

* stage_WB discards the register write

* earlier pipeline stages still perform the access/operation

* but do not assume it ends a trap shadow in PAL modeling

 

4. Specific Instruction-Class Rules Involving R31

 

4.1 Branch and Jump Link Semantics with R31 as Ra

 

Rule R31-BR-1:

BR/BSR and JMP/JSR/RET/JSR_COROUTINE execute normally when Ra=R31, but no link

value (return PC) can be saved because the destination is discarded.

 

Pipeline implication:

 

* Control transfer proceeds normally.

* Any "link" writeback to Ra is suppressed by stage_WB due to R31-DST-1.

 

Qualifiable behavior:

 

* Treat link-write to R31 exactly as any other write to R31: discard.

 

4.2 Atomic Reservation Instructions: LDx_L / STx_C with R31 as destination

 

 

* STx_C R31, disp(Rb): lock_flag and location become UNPREDICTABLE.

* LDx_L R31, disp(Rb): lock_flag and locked_physical_address become  UNPREDICTABLE.

 

Rule R31-ATOMIC-1:

If LDx_L targets R31, the reservation state becomes UNPREDICTABLE.

Rule R31-ATOMIC-2:

If STx_C targets R31, the reservation state and location become UNPREDICTABLE.

 

Emulator must choose a deterministic policy. Recommended policy for a qualifiable emulator:

 

Policy ATOMIC-R31-DET (Deterministic, conservative, safe):

A) LDx_L with Rc=31:

- Perform the load's address translation and memory read exactly as normal (so faults still occur normally).

- Do NOT create or update a reservation (do not set lock_flag and do not set locked_physical_address).

- Discard loaded value (since Rc=31).

- Record a trace/counter: "LDx_L to R31 encountered".

 

B) STx_C with Rc=31:

- Perform address translation exactly as normal (so faults still occur).

- Do NOT perform the conditional store (treat as "failed conditional").

- Invalidate any existing reservation for this CPU (clear lock_flag).

- Discard success/failure result (since Rc=31).

- Record a trace/counter: "STx_C to R31 encountered".

 

Rationale:

 

•Prevents corrupting shared locations.

•Avoids fabricating reservation state that the ISA calls UNPREDICTABLE.

•Preserves fault behavior and pipeline ordering.

•Produces stable behavior for debugging and qualification.

 

Important:

This policy must be documented as an emulator-defined resolution of

UNPREDICTABLE behavior.

 

4.3 "Must be R31" operand constraints (CTLZ/CTPOP/CTTZ, SEXTx, etc.)

 

Your notes include constraints such as:

 

* CTLZ/CTPOP/CTTZ: Ra must be R31.

* SEXTx: Ra must be R31.

* ITOFx: Rb must be R31.

* AMASK: Ra must be R31 or result is UNPREDICTABLE.

* IMPLVER: Ra must be R31 and Rb must be literal #1 or UNPREDICTABLE.

 

Two valid emulator approaches exist:

 

Approach A (Strict, diagnostic-friendly):

If a "must be R31" constraint is violated, raise an Illegal Operand / Reserved Instruction style exception via FaultDispatcher.

 

Approach B (Literal ISA wording, UNPREDICTABLE):

If violated, produce an arbitrary result and possibly arbitrary exception behavior.

 

Recommended for qualification:

Use Approach A for deterministic behavior, unless you have evidence that the target firmware/OS depends on UNPREDICTABLE outcomes (unlikely).

Document that this is an emulator-defined tightening of operand constraints.

 

Pipeline implication:

 

* These checks should occur during decode or execute, not WB.

* WB still discards writes to R31 where applicable.

 

4.4 Logical idioms using R31 as constant zero

 

Examples from your notes:

 

* ORNOT with Ra=R31 performs NOT.

* BIS R31, R31, Rx clears Rx (CLR).

* CMPBGE R31, Rb, Rc uses zero for Ra.

 

Rule R31-LOGIC-1:

Treat R31 reads as 0; these idioms naturally fall out.

 

No special WB action beyond normal "discard if Rc==31".

 

4.5 Prefetch / Hint idioms using R31 (and F31)

 

Your notes include prefetch idioms:

 

* LDL R31, disp(Rb)  (prefetch)

* LDQ R31, disp(Rb)  (prefetch enable)

* LDS F31, disp(Rb)  (prefetch M)

* LDT F31, disp(Rb)  (prefetch M enable)

 

Rule R31-PREF-1 (Integer side):

Loads to R31 may be treated as hints and "may not be executed" by some

implementations. Exceptions are still always signaled for instruction fetch.

 

Emulator policy recommendation:

For now (as you requested: "let PAL handle complexity"):

- Execute loads normally for correctness.

- Discard destination write due to R31.

- Add trace counters so you can later decide whether to convert these into

non-faulting prefetch behavior for performance.

 

5. AlphaPipeline / stage_WB Requirements

 

 

5.1 Core WB rule

 

WB-R31-1:

If slot.destIntReg == 31 then:

- Do not write to the integer register file

- Do not mark reg31 as "dirty"

- Do not generate reg-write trace events (except optional "discarded write"

diagnostics)

 

WB-R31-2:

Source reads of register 31 must never take dependencies and must always read

0.

 

5.2 Link writeback handling

 

WB-R31-3:

For branch/jump with link destination Ra:

- If Ra==31, discard link write.

- Control transfer still commits normally (PC update is not suppressed).

 

5.3 Reservation/atomic integration point

 

WB-R31-4:

Reservation state (lock_flag / locked_physical_address) must NOT be updated

in stage_WB based on a destination register write.

Reservation state is a side effect of the memory operation itself and must

be committed by the memory/atomic stage.

 

This is important for the R31 atomic UNPREDICTABLE cases:

 

* The atomic stage must detect Rc==31 and apply policy ATOMIC-R31-DET.

* WB simply discards the (already meaningless) result.

 

5.4 Exception / fault sequencing

 

WB-R31-5:

WB does not suppress exceptions. If earlier stages produced a PendingEvent,

WB must not "complete" a discarded register write.

 

(Your "do nothing now; PAL handles it" plan aligns with this.)

 

6. Initialization Rules

 

INIT-R31-1:

On CPU reset, R31 reads as 0.

 

INIT-R31-2:

R31 must remain 0 for the lifetime of the CPU, because writes are discarded.

 

Implementation best practice:

 

* Do not store a backing value for R31 at all.

* readIntReg(31) returns 0.

* writeIntReg(31, x) is a no-op.

 

7. Trace/Profiling Requirements (recommended)

 

Since you intend to let PAL handle alignment/complexity initially, add low-cost

counters so you can profile and revisit later.

 

Counters (per CPU recommended):

 

* r31_discarded_writes_total

* r31_discarded_link_writes

* r31_atomic_ldl_to_r31

* r31_atomic_stc_to_r31

* r31_prefetch_loads_to_r31

* r31_operand_constraint_violations (if you implement Approach A)

 

Event logging:

 

* Keep it conditional (compile-time or runtime flag)

* Avoid logging in the hot path unless tracing enabled

 

8. Summary of Qualifiable Emulator Policy

 

* R31 reads supply 0 always.

* R31 writes are always discarded in stage_WB.

* Branch/jump with Ra=R31 works, link value is discarded.

* LDx_L/STx_C with Rc=31 are architecturally UNPREDICTABLE; emulator chooses a

 documented deterministic behavior (ATOMIC-R31-DET) that is safe and stable.

* "Ra must be R31" style constraints: prefer deterministic faulting (Approach A)

 for qualification and debugging.

* Prefetch idioms using loads to R31: execute normally now, add profiling.

 

9. Implementation Checklist for a Dedicated R31 Handler

 

To build a "qualifiable R31 handler" for AlphaPipeline:

 

[ ] Provide readIntReg(31) -> 0 fast-path (no backing storage needed).

[ ] Provide writeIntReg(31, x) -> no-op.

[ ] In stage_WB:

- if dest==31: skip writeback

- if linkDest==31: skip link write

- still commit PC updates and other side effects

[ ] In atomic memory stage:

- detect LDx_L/STx_C with Rc==31 and apply ATOMIC-R31-DET policy

[ ] Add trace counters for discarded writes and R31-atomic cases

[ ] Document operand-constraint enforcement choice (Approach A recommended)

 

10. Notes on F31 (non-normative here)

 

F31 rules are more severe than R31:

 

•F31 as source supplies "true zero" (see Section 4.7.3).

•F31 as destination discards results AND it is UNPREDICTABLE whether other destination operands are changed or whether exceptions are signaled for many operations.

•Loads to F31 do not signal MM/alignment exceptions; FP disabled exception is  UNPREDICTABLE.
