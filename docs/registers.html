Registers

CBox operations participate in the architectural order of operations as serialized, atomic control commits; the MSB-first 6-bit shifting is purely microarchitectural and must be collapsed into a single grain-level state transition that enforces the same ordering guarantees as PAL entry, interrupts,

and HW_REI.

 

"Order is guaranteed by program-order retirement and explicit serialization points; any asynchronous backends MUST be drained/fenced at those points."

 

•The EmulatR implements deterministic retirement order of the PC+n / serialized grain loop.

•The CBox ordering implementation is not only "program order"; it also requires a serialization point that fences visibility of certain effects relative to:

ointerrupts / PAL entry,

oinstruction fetch coherency (I-stream),

oand any microarchitectural concurrency you simulate (async mem, device DMA, etc.).

 

A.Grain as the architectural "retirement event"

An instruction grain represents a single Alpha instruction at the point it is architecturally retired. The AlphaPipeline executes grains in program order and commits each grain's architected side effects before advancing the PC to the next instruction (PC+4 or redirected PC).

 

B. CBox grains are serialization grains

CBox-related grains (e.g., hw_mtpr_I_CTL, hw_mtpr_IC_FLUSH, hw_mfpr_I_CTL) are modeled as "serialized control transactions". The grain does not model EV6 microarchitectural mechanisms (e.g., internal shift-register loading of 6-bit chunks into C_DATA). Instead, the grain models the architectural contract:

•The operation is atomic with respect to architectural observation.

•All prior instructions are complete before the CBox state change becomes visible.

•No subsequent instruction observes a partial CBox update.

 

C. What "complete" means for CBox retirement

At retirement of a CBox grain, the EmulatR guarantees:

 

1.The requested CBox control effect is logically committed (or treated as a no-op with preserved ordering, where appropriate).

2.The pipeline is serialized such that interrupts cannot be taken "mid-CBox".

3.Any required instruction-stream ordering point is established (e.g., an IC_FLUSH acts as an I-stream barrier even if the emulator does not model a physically stale I-cache).

 

D. Where the ordering comes from

The ordering semantics are achieved by:

 

•The AlphaPipeline's in-order retirement (grain pumping), AND

•An explicit "serialize pipeline" step performed by CBox grains to ensure that side effects are completed and visible before PC advances.

•The AlphaPipeline  enforces "retire one grain at a time and complete all its side effects before PC advances"; the AlphaPipleline provides the essential architectural ordering property that CBox needs.

 

Order of Operation Rule for CBox Grains.

Include these as short "MUST" statements in your documentation:

 

Rule 1: Grain retirement is the architectural commit boundary.

Rule 2: CBox grains perform a serialization step before commit.

Rule 3: Interrupt delivery checks occur only at grain boundaries or at explicit pipeline safe points; serialization grains block mid-grain interrupt entry.

Rule 4: MB/WMB and CBox serialization drain/fence any emulator-internal async queues that can reorder visibility (SafeMemory, MMIO, device I/O).

Rule 5: IC_FLUSH may be modeled as a no-op only if it still provides the ordering point required for I-stream correctness (e.g., invalidating translated code cache entries for the affected region if you have a JIT/cache).

 

Grain Order of Operation (PC+n) and AlphaPipeLine with MB/WMB fencing handle the following C_Box bit shifting:

 

CBox Bit Shifting Operations:

Split into three shift register chains.

1.The hardware allocates 367 bits for the /write_ONCE chain, of which the 21264 uses 303 bits.  During hardware reset (after BiST), 367 bits are always shifted into the WRITE_ONCE chain from the SROM, MSB first, so that anu unused bits are shifted out the end of the WRITE_ONCE chain.

2.A 36-bit WRITE_MANY chain that is loaded using MTPR instructions to the CBox data register.

a.Six bits of information are shifted into the WRITE_MANY chain during each write transaction to the CBox data register.

3.A 60-bit CBox ERROR_REG chain that is read by using MFFR instructions from the Cbox data register in combination with MTPR instruction to the CBox shift register. Each write transaction to the CBox shift register destructively shifts six bits of information out of the Cbox error register.
